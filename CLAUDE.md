## What is Sārathi?

Sārathi (सारथी) is a silent AI co-pilot for BrightChamps teachers during live 1:1 coding classes. Named after Krishna — the charioteer who whispered strategy to Arjuna without ever taking the battlefield himself.

It is a desktop overlay app (Electron). The teacher presses a hotkey at any moment during class — on the lesson slides, on code.org, on Sprite Lab, anywhere — and Sārathi:

1. Takes a silent screenshot of whatever is on screen
2. Sends it to Claude vision with the loaded lesson as context
3. Shows a floating panel with 3 smart options + open text field
4. Streams a response grounded in the lesson
5. Draws an orange circle annotation on the teacher's screen if relevant

**Students never see it. It never speaks. It never leaves the teacher's screen.**

## Core files — what each one does

### `src/main.js`
Electron main process. Responsible for:
- App lifecycle (no dock icon: `app.dock.hide()`)
- Global hotkey registration: `Cmd+Shift+Space` (Mac) / `Ctrl+Shift+Space` (Win)
- Silent screenshot via `desktopCapturer`
- Creating and managing two BrowserWindows:
  - `launcherWindow` — small lesson selector shown at startup, then hidden
  - `overlayWindow` — fullscreen transparent always-on-top click-through overlay
- IPC bridge between main process and overlay renderer
- Zoom exclusion: overlay must never appear in screen shares

### `src/sarathi.js`
The brain. Responsible for:
- Loading lesson JSON from `lessons/`
- Building the Claude system prompt (role + lesson context + grounding rules + annotation instructions)
- Sending screenshot (base64) + lesson context to Anthropic API (`claude-sonnet-4-6`)
- Parsing Claude response: extracting 3 options, explanation text, `[CIRCLE:x,y,r]` tags
- Exposing `generateOptions(screenshotBase64, lessonJson)` and `askFollowup(question, screenshotBase64, lessonJson)`

### `src/overlay.html`
Single-file UI. Contains:
- Prompt panel: 3 option buttons + open text field (shown after hotkey)
- Response card: streaming text display below the panel
- Annotation canvas: fullscreen transparent canvas for circle drawing
- IPC listeners: receives `show-panel`, `show-response`, `draw-circle`, `clear` from main
- All CSS inline — no external stylesheets, no frameworks
- Keyboard: `1`/`2`/`3` select options, `Enter` submits, `Escape` dismisses

## The demo we are building toward

**This is the definition of done for MVP:**

1. Teacher has Sārathi running (tray icon visible, no dock icon)
2. Lesson 1 (Fun Rainbow) is loaded
3. Teacher is on the lesson slides — presses `Cmd+Shift+Space`
4. Panel appears in top-right within 2 seconds, 3 relevant options visible
5. Teacher clicks an option — response streams in
6. Teacher switches to code.org tab — presses hotkey again
7. New panel appears with options relevant to the code on screen
8. Teacher types a question in open field — submits
9. Response appears + orange circle drawn on the incorrect element
10. Circle fades after 8 seconds
11. Overlay is invisible on Zoom screen share (students see nothing)

**Record this in one take. That is the MVP.**

## Technical constraints

### Zoom screen share exclusion — CRITICAL
The overlay window MUST be excluded from Zoom and all screen capture tools.

**Mac:**
```javascript
overlayWindow.setContentProtection(true)
// or via native: NSWindowSharingNone
```

**Windows:**
```javascript
// Use SetWindowDisplayAffinity with WDA_EXCLUDEFROMCAPTURE
// Via electron: overlayWindow.setContentProtection(true)
```

Test this before anything else. If students can see the overlay, the product is broken.

### Click-through overlay
The overlay must never block the teacher from interacting with apps beneath it.

```javascript
overlayWindow.setIgnoreMouseEvents(true, { forward: true })
// Temporarily disable click-through when panel is shown:
overlayWindow.setIgnoreMouseEvents(false)
// Re-enable when panel is dismissed
```

### Screenshot approach
Use `desktopCapturer` not `screenshot` npm packages — built into Electron, no extra permissions dialogs.

```javascript
const sources = await desktopCapturer.getSources({
  types: ['screen'],
  thumbnailSize: { width: 1920, height: 1080 }
})
const screenshot = sources[0].thumbnail.toDataURL()
```

### API key
Loaded from `.env` via `dotenv`. Never hardcoded. Never committed.
```
ANTHROPIC_API_KEY=sk-ant-...
```

## Claude system prompt template

This is the exact system prompt structure `sarathi.js` assembles for every API call:

```
You are Sārathi, a silent AI co-pilot for a BrightChamps coding teacher in a live class.

LESSON LOADED:
Title: {lesson_title}
Topics: {topics_covered}
Objective: {objective}
Platform: {tool}
Duration: {duration_mins} minutes

LESSON CONCEPTS:
{concept_definitions}

LESSON Q&A (warmup + checkpoints):
{warmup_qa}
{checkpoint_qa}

COMMON ERRORS IN THIS LESSON:
{common_errors}

YOUR ROLE:
- You help the teacher, not the student. Use teacher language.
- Ground every response in the lesson content above. Do not invent.
- If the question is outside this lesson's scope, say: "That's outside today's lesson scope."
- Be direct. 2–4 sentences maximum. No padding.
- If your response references a specific element visible on screen, end with [CIRCLE:x,y,r] where x,y are the pixel coordinates of the element's center and r is an appropriate radius in pixels.

FIRST TASK:
Look at the screenshot. Identify what is visible (slide, code editor, browser, etc.).
Generate exactly 3 short prompt suggestions the teacher might want to ask right now.
Each suggestion must be max 8 words. Make them specific to what's on screen.

Format your response as:
OPTIONS:
1. {option 1}
2. {option 2}
3. {option 3}
```

## Annotation system

Claude embeds tags in its responses. `sarathi.js` strips these before displaying text to the teacher.

| Tag | What gets drawn | Example |
|-----|----------------|---------|
| `[CIRCLE:x,y,r]` | Orange circle on overlay canvas | `[CIRCLE:640,380,45]` |

MVP uses only `[CIRCLE]`. Future tags (not built yet): `[ARROW:x1,y1,x2,y2]`, `[TEXT:x,y,content]`, `[UNDERLINE:x1,y1,x2,y2]`.

Circle spec:
- Color: `#E8621A` (BrightChamps orange)
- Stroke width: 3px
- Style: slightly dashed (`setLineDash([8, 4])`) for pencil feel
- Fade: visible 7 seconds, fade over 1 second

## Code style

- **Clarity over cleverness.** A junior developer should understand every function from its name alone.
- **No frameworks.** Vanilla JS only. No React, no Vue, no bundlers. This is a 3-file app.
- **No npm packages beyond Electron and dotenv.** Every dependency is a liability in a local demo app.
- **Async/await throughout.** No callbacks except where Electron APIs require them.
- **IPC naming convention:** `kebab-case` event names. e.g. `show-panel`, `draw-circle`, `clear-overlay`.
- **Comments explain WHY, not WHAT.** If the code is clear, don't comment it.
- **Error handling:** Every API call gets a try/catch. Log errors to console. Show "Sārathi couldn't reach the AI — check your connection." in the panel if the call fails.

## What NOT to do

- Do NOT add voice input or TTS
- Do NOT add authentication or login
- Do NOT add session scheduling or class ID validation
- Do NOT add multi-monitor support
- Do NOT use React, Vue, or any UI framework
- Do NOT use external CSS frameworks (Tailwind, Bootstrap)
- Do NOT add arrow/underline/text annotations (circle only in MVP)
- Do NOT run `npm start` or `npx electron .` — always tell the human to do this
- Do NOT commit `.env` or any file containing `ANTHROPIC_API_KEY`
- Do NOT make the overlay visible to screen share before testing `setContentProtection`
